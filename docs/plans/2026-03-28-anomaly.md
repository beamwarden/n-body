# Implementation Plan: Anomaly Detection and Classification
Date: 2026-03-28
Status: Approved

## Summary

Implement the six stub functions in `backend/anomaly.py` to provide NIS-based anomaly
detection, three-way anomaly classification (maneuver, drag anomaly, filter divergence),
recalibration triggering, and SQLite persistence of anomaly events with recalibration
duration tracking. This plan also replaces the `pytest.skip` stubs in
`tests/test_anomaly.py` with working assertions and adds additional tests for full
coverage of the classification logic and database operations.

## Requirements addressed

- **F-030 [DEMO]**: Flag anomaly when NIS exceeds chi-squared critical value (p=0.05, 6 DOF).
- **F-031 [DEMO]**: Classify detected anomalies into `maneuver`, `drag_anomaly`, `filter_divergence`.
- **F-032**: Maneuver classification requires NIS elevation on at least 2 consecutive update cycles for an active satellite.
- **F-033 [DEMO]**: Trigger filter recalibration when anomaly detected (re-init state from new observation with inflated covariance).
- **F-034**: Record time between anomaly detection and recalibration completion (return to normal NIS).
- **F-035**: Store all anomaly events: NORAD ID, detection epoch, anomaly type, NIS value, recalibration duration.

## Files affected

- `backend/anomaly.py` -- Implement all six stubs: `evaluate_nis`, `classify_anomaly`,
  `trigger_recalibration`, `record_anomaly`, `record_recalibration_complete`,
  `get_active_anomalies`. Add a module-level function `ensure_alerts_table` for SQLite
  schema creation. Add a private helper `_count_consecutive_tail_exceedances` for
  maneuver classification logic.
- `tests/test_anomaly.py` -- Replace all six `pytest.skip` stubs with working assertions.
  Add additional tests for drag anomaly classification, edge cases (empty NIS history,
  boundary NIS values), database round-trip, and recalibration duration computation.
- `backend/main.py` -- No changes in this plan. The integration of anomaly.py into the
  main update loop (calling `classify_anomaly` after each `kalman.update`, broadcasting
  anomaly WebSocket messages) will be addressed in a separate API/integration plan.
  This plan delivers anomaly.py as a standalone, fully tested module.

## Data flow changes

**Before:** All `anomaly.py` functions raise `NotImplementedError`. No anomaly detection,
classification, or persistence occurs.

**After:**
```
kalman.update() returns filter_state dict
        |
        v
filter_state['nis']               --> anomaly.evaluate_nis(nis, threshold)
filter_state['nis_history']       --> |
filter_state['innovation_eci_km'] --> anomaly.classify_anomaly(
filter_state['anomaly_flag']      -->     norad_id, nis_history, innovation_eci_km,
                                          is_active_satellite, threshold)
        |
        v  (if anomaly_type is not None)
anomaly.record_anomaly(db, norad_id, epoch_utc, anomaly_type, nis_value)
        --> returns anomaly_row_id (stored for later resolution)
        |
        v
anomaly.trigger_recalibration(norad_id, anomaly_type, epoch_utc)
        --> returns recalibration_params dict
        --> caller passes to kalman.recalibrate()
        |
        v  (after subsequent updates, when NIS returns below threshold)
anomaly.record_recalibration_complete(db, anomaly_row_id, resolution_epoch_utc)
        --> computes and stores recalibration_duration_s (F-034)
```

**Inputs consumed from kalman.py filter_state dict** (schema defined in Kalman plan):
- `filter_state['nis']` -- float, current NIS value
- `filter_state['nis_history']` -- list[float], all past NIS values (most recent last)
- `filter_state['innovation_eci_km']` -- NDArray[float64] shape (6,), innovation vector
- `filter_state['anomaly_flag']` -- bool, True when NIS > CHI2_THRESHOLD_6DOF

**Outputs produced:**
- Anomaly classification string (or None) for the caller to include in WebSocket messages
- Recalibration parameter dict for the caller to pass to `kalman.recalibrate()`
- SQLite rows in the `alerts` table for persistence and query by API endpoints

## Implementation steps

### Phase 1: SQLite schema and table creation

1. **Add `ensure_alerts_table` function** (`backend/anomaly.py`)
   - Action: Add a new public function after the existing constants block:
     ```
     def ensure_alerts_table(db: sqlite3.Connection) -> None:
     ```
     This function executes `CREATE TABLE IF NOT EXISTS alerts` with the following columns:
     - `id` INTEGER PRIMARY KEY AUTOINCREMENT
     - `norad_id` INTEGER NOT NULL
     - `detection_epoch_utc` TEXT NOT NULL (ISO-8601 UTC string)
     - `anomaly_type` TEXT NOT NULL (one of 'maneuver', 'drag_anomaly', 'filter_divergence')
     - `nis_value` REAL NOT NULL
     - `resolution_epoch_utc` TEXT (nullable -- NULL until recalibration completes)
     - `recalibration_duration_s` REAL (nullable -- computed on resolution)
     - `status` TEXT NOT NULL DEFAULT 'active' (values: 'active', 'recalibrating', 'resolved')
     - `created_at` TEXT NOT NULL DEFAULT (current timestamp in UTC)

     The function also creates an index on `(norad_id, status)` for efficient queries by
     `get_active_anomalies`.

     The database path is determined by the caller (expected to come from
     `os.environ.get("NBODY_DB_PATH", "data/catalog/tle_cache.db")` in `main.py`).
     `anomaly.py` receives an already-open `sqlite3.Connection` -- it does not open
     connections itself.
   - Why: F-035 requires persistent storage of anomaly events. The schema must exist
     before any `record_anomaly` calls. Placing table creation in a dedicated function
     allows `main.py` to call it at startup.
   - Dependencies: None.
   - Risk: **Low**

### Phase 2: Core detection and classification logic

2. **Implement `evaluate_nis`** (`backend/anomaly.py`)
   - Action: Replace the `raise NotImplementedError` body with:
     - Return `nis > threshold`
     - The default threshold `12.592` is the chi-squared critical value for 6 DOF at
       p=0.05, matching `kalman.CHI2_THRESHOLD_6DOF`. Import and reference the constant
       from `kalman.py` rather than defining a second copy:
       ```
       from backend.kalman import CHI2_THRESHOLD_6DOF
       ```
       Update the function default to `threshold: float = CHI2_THRESHOLD_6DOF`.
     - Validate that `nis` is non-negative. If `nis < 0`, raise `ValueError`.
   - Why: F-030 requires NIS threshold comparison. This is intentionally a thin function
     because the threshold is a single comparison, but wrapping it gives tests a stable
     interface and lets the threshold be overridden.
   - Dependencies: None.
   - Risk: **Low**

3. **Add `_count_consecutive_tail_exceedances` private helper** (`backend/anomaly.py`)
   - Action: Add a private function:
     ```
     def _count_consecutive_tail_exceedances(
         nis_history: list[float],
         threshold: float,
     ) -> int:
     ```
     Starting from the end of `nis_history`, count how many consecutive values exceed
     `threshold`. Return that count. If the list is empty, return 0.

     Example: `[3.0, 15.0, 14.0]` with threshold 12.592 returns 2 (the last two values
     exceed threshold). `[15.0, 3.0, 14.0]` returns 1 (only the last value).
   - Why: F-032 requires counting consecutive elevated NIS cycles. Extracting this into
     a helper keeps `classify_anomaly` readable and makes the counting logic independently
     testable.
   - Dependencies: None.
   - Risk: **Low**

4. **Implement `classify_anomaly`** (`backend/anomaly.py`)
   - Action: Replace the `raise NotImplementedError` body with the following classification
     logic, applied in priority order (first match wins):

     a. **Check for anomaly presence:** Call `evaluate_nis(nis_history[-1], threshold)` on
        the most recent NIS value. If `nis_history` is empty or the most recent NIS is
        below threshold, return `None` (no anomaly).

     b. **Maneuver check (F-032):** Call `_count_consecutive_tail_exceedances(nis_history,
        threshold)`. If the count is >= 2 AND `is_active_satellite` is True, return
        `ANOMALY_MANEUVER`.

     c. **Drag anomaly check:** Examine `innovation_eci_km`. Compute the along-track
        residual magnitude as `abs(innovation_eci_km[0])` and the cross-track residual
        magnitude as `sqrt(innovation_eci_km[1]**2 + innovation_eci_km[2]**2)`.

        **Important note on frame simplification:** The innovation vector is in ECI, not
        RSW (radial, along-track, cross-track). A proper along-track vs. cross-track
        decomposition requires the velocity vector to define the RSW frame. For POC, we
        use a simplified heuristic: the first three components of the innovation vector
        (position residual) are examined. If the magnitude of the position residual
        projected onto the velocity direction dominates the perpendicular component by a
        factor of 3:1 or more, classify as drag anomaly. Specifically:

        - Compute `pos_residual_km = innovation_eci_km[0:3]`
        - Compute `vel_direction = innovation_eci_km[3:6]`; if `np.linalg.norm(vel_direction)
          < 1e-10`, skip drag classification (velocity residual too small to determine
          direction).
        - Otherwise, `vel_unit = vel_direction / np.linalg.norm(vel_direction)`
        - `along_track_km = abs(np.dot(pos_residual_km, vel_unit))`
        - `cross_track_km = np.linalg.norm(pos_residual_km - np.dot(pos_residual_km,
          vel_unit) * vel_unit)`
        - If `along_track_km > 3.0 * cross_track_km` and `cross_track_km < 1.0` (km),
          return `ANOMALY_DRAG`.

        **Design note:** This heuristic uses the velocity *residual* direction as a proxy
        for the along-track direction. This is imperfect but sufficient for POC
        demonstration purposes. A production implementation would use the object's actual
        velocity vector from the filter state to define the RSW frame. The implementer
        should add a code comment noting this simplification.

     d. **Fallback -- filter divergence:** If none of the above matched but the most recent
        NIS exceeds threshold, return `ANOMALY_DIVERGENCE`.

   - Why: F-031 requires three-way classification. The priority order (maneuver first, then
     drag, then divergence as catch-all) ensures that the most operationally significant
     events are classified specifically, with divergence as a safe fallback.
   - Dependencies: Requires steps 2 and 3.
   - Risk: **Medium**. The drag anomaly heuristic using velocity residual as along-track
     proxy is a deliberate simplification. It may misclassify some events. This is
     acceptable for POC but should be documented clearly in the code.

### Phase 3: Recalibration trigger

5. **Implement `trigger_recalibration`** (`backend/anomaly.py`)
   - Action: Replace the `raise NotImplementedError` body with:
     - Validate that `epoch_utc` is UTC-aware. If `epoch_utc.tzinfo is None`, raise
       `ValueError("epoch_utc must be UTC-aware")`.
     - Validate that `anomaly_type` is one of the three constants (`ANOMALY_MANEUVER`,
       `ANOMALY_DRAG`, `ANOMALY_DIVERGENCE`). If not, raise `ValueError`.
     - Determine `inflation_factor` based on anomaly type:
       - `maneuver`: 20.0 (large inflation because the orbit has physically changed)
       - `drag_anomaly`: 10.0 (moderate inflation for model mismatch)
       - `filter_divergence`: 10.0 (default)
     - Return a dict:
       ```python
       {
           "norad_id": norad_id,
           "anomaly_type": anomaly_type,
           "epoch_utc": epoch_utc,
           "inflation_factor": inflation_factor,
           "status": "pending",
       }
       ```
   - Why: F-033 requires recalibration triggering. The returned dict is designed to be
     passed directly to `kalman.recalibrate(filter_state, observation_eci_km, epoch_utc,
     inflation_factor=params["inflation_factor"])` by the caller. The anomaly-type-dependent
     inflation factor encodes the physical intuition that a maneuver represents a larger
     state discontinuity than a drag model error.
   - Dependencies: None.
   - Risk: **Low**. The inflation factors are hand-tuned heuristics. They can be adjusted
     without architectural changes.

### Phase 4: Database persistence

6. **Implement `record_anomaly`** (`backend/anomaly.py`)
   - Action: Replace the `raise NotImplementedError` body with:
     - Validate `detection_epoch_utc` is UTC-aware. Raise `ValueError` if not.
     - Execute an INSERT into the `alerts` table:
       ```sql
       INSERT INTO alerts (norad_id, detection_epoch_utc, anomaly_type, nis_value, status)
       VALUES (?, ?, ?, ?, 'active')
       ```
       Where `detection_epoch_utc` is formatted as ISO-8601 string via
       `detection_epoch_utc.isoformat()`.
     - Commit the transaction.
     - Return `cursor.lastrowid`.
   - Why: F-035 requires persistent anomaly event storage.
   - Dependencies: Requires step 1 (table must exist).
   - Risk: **Low**

7. **Implement `record_recalibration_complete`** (`backend/anomaly.py`)
   - Action: Replace the `raise NotImplementedError` body with:
     - Validate `resolution_epoch_utc` is UTC-aware. Raise `ValueError` if not.
     - SELECT `detection_epoch_utc` from the `alerts` table for the given `anomaly_row_id`.
       If no row found, raise `ValueError(f"No anomaly record with id {anomaly_row_id}")`.
     - Parse the stored `detection_epoch_utc` string back to a datetime.
     - Compute `recalibration_duration_s = (resolution_epoch_utc -
       detection_epoch_utc).total_seconds()`.
     - UPDATE the row:
       ```sql
       UPDATE alerts SET
         resolution_epoch_utc = ?,
         recalibration_duration_s = ?,
         status = 'resolved'
       WHERE id = ?
       ```
     - Commit the transaction.
   - Why: F-034 requires recording time-to-resolution. F-035 requires storing
     recalibration duration.
   - Dependencies: Requires steps 1 and 6.
   - Risk: **Low**. Parsing ISO-8601 strings back to datetime requires care with timezone
     info. Use `datetime.datetime.fromisoformat()` (available in Python 3.11+) which
     handles the `+00:00` suffix. The implementer must ensure the stored string preserves
     timezone info.

8. **Implement `get_active_anomalies`** (`backend/anomaly.py`)
   - Action: Replace the `raise NotImplementedError` body with:
     - SELECT all rows from `alerts` where `status != 'resolved'`.
     - Return a list of dicts, each containing: `id`, `norad_id`, `detection_epoch_utc`,
       `anomaly_type`, `nis_value`, `status`.
     - Parse `detection_epoch_utc` from ISO-8601 string back to `datetime.datetime` in
       each returned dict.
   - Why: This is needed by the API layer (`GET /catalog` enrichment, alerts panel) and
     by the main update loop to track which objects are currently in anomaly state.
   - Dependencies: Requires step 1.
   - Risk: **Low**

## Risks and mitigations

- **Risk:** The drag anomaly heuristic uses the velocity *residual* direction as a proxy
  for the along-track direction, which is physically imprecise. -- **Mitigation:** This
  is a known POC simplification. Add a clear code comment and a TODO referencing the
  proper RSW decomposition for post-POC. For the demo, drag anomaly events are secondary
  to maneuver detection (the "funding moment").

- **Risk:** The `classify_anomaly` function receives `is_active_satellite` as a boolean
  parameter, but there is no catalog metadata source defined yet that provides this flag.
  -- **Mitigation:** The caller (expected to be `main.py`) is responsible for looking up
  the object's catalog classification. For POC, this can be a static dict or a column in
  the SQLite catalog table. This is an integration concern outside this plan's scope but
  must be addressed before end-to-end demo.

- **Risk:** Storing `detection_epoch_utc` as an ISO-8601 string in SQLite and parsing it
  back could lose timezone information if not handled carefully. -- **Mitigation:** The
  implementer must always store with `datetime.isoformat()` (which preserves `+00:00` for
  UTC-aware datetimes) and parse with `datetime.fromisoformat()`. Add a test that
  round-trips a UTC-aware datetime through the database and verifies timezone is preserved.

- **Risk:** The `import` of `CHI2_THRESHOLD_6DOF` from `backend.kalman` creates a module
  dependency. If `kalman.py` has heavy imports (numpy, filterpy) this increases
  `anomaly.py` import time. -- **Mitigation:** The constant is a plain float. If import
  weight becomes a concern, duplicate the constant in `anomaly.py` with a comment
  referencing the source. For POC, the cross-import is acceptable.
