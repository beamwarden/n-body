# Implementation Plan: API and WebSocket Gateway (main.py)
Date: 2026-03-28
Status: Approved

## Summary

Implement the FastAPI REST endpoints (`GET /catalog`, `GET /object/{norad_id}/history`) and the WebSocket streaming endpoint (`/ws/live`) in `backend/main.py`. This plan also specifies the concurrency model: how `main.py` wires the ingest polling loop, Kalman filter updates, anomaly detection, and WebSocket broadcast into a single asyncio event loop using background tasks and an in-process `asyncio.Queue` event bus. The plan additionally creates the SQLite `state_history` table (required by F-041) and fills in the test file `tests/test_main.py`.

## Requirements addressed

- **F-040** `GET /catalog` -- list tracked objects with current state summary
- **F-041** `GET /object/{norad_id}/history` -- time-series of state updates and NIS values
- **F-042 [DEMO]** WebSocket `/ws/live` -- real-time state update and anomaly alert push
- **F-043** WebSocket messages conform to architecture Section 3.5 JSON schema
- **F-044** WebSocket supports at least 5 simultaneous connections without degradation
- **NF-002** WebSocket broadcast latency under 500ms end-to-end on local network
- **NF-012** Reconnecting WebSocket client receives current state within 5 seconds

## Files affected

- `backend/main.py` -- replace all `NotImplementedError` stubs with working implementations; add lifespan startup/shutdown logic, WebSocket connection manager, background processing task, and REST endpoint bodies
- `tests/test_main.py` -- replace `pytest.skip` stubs with real tests using FastAPI TestClient and WebSocket test client
- `backend/ingest.py` -- no changes (read only; event bus interface via `asyncio.Queue` already exists in `poll_once` and `run_ingest_loop`)
- `backend/kalman.py` -- no changes (read only; `get_state()` return schema already documented)
- `backend/anomaly.py` -- no changes (read only; `classify_anomaly()` and `get_active_anomalies()` signatures already defined)

## Data flow changes

### Before (stub state)

No data flows. All endpoints raise `NotImplementedError`. No background tasks. No event bus wiring.

### After (implemented)

```
                  lifespan startup
                        |
        +---------------+----------------+
        |               |                |
  init_catalog_db  load_catalog_config  create asyncio.Queue (event_bus)
        |               |                |
        +-------+-------+----------------+
                |
      launch background task: _ingest_loop_task
      launch background task: _processing_loop_task
                |
                v
  _ingest_loop_task:
      calls ingest.run_ingest_loop(db_path, catalog_config_path, event_bus)
      (this runs forever, putting 'catalog_update' events on event_bus)
                |
                v
  _processing_loop_task:
      awaits event_bus.get()
      on 'catalog_update':
          for each norad_id in catalog:
              get_latest_tle(db, norad_id)
              if no filter exists: init_filter from TLE
              else: predict + update (kalman.predict, kalman.update)
              classify_anomaly (anomaly.classify_anomaly)
              if anomaly: record_anomaly, trigger_recalibration, recalibrate filter
              store state snapshot to state_history table
              build WebSocket message (architecture 3.5 schema)
              broadcast to all connected WebSocket clients

  GET /catalog:
      reads from in-memory filter_states dict
      returns [{norad_id, name, last_update_epoch_utc, confidence}]

  GET /object/{norad_id}/history:
      reads from state_history SQLite table
      returns [{epoch_utc, eci_km, eci_km_s, nis, confidence}]

  WebSocket /ws/live:
      accepts connection, adds to connection_manager set
      on connect: sends current state of all objects (NF-012)
      loops receiving (keepalive); removes on disconnect
```

## Implementation steps

### Phase 1: Application state and database schema

1. **Add `state_history` table creation** (`backend/main.py`)
   - Action: Add a module-level function `_ensure_state_history_table(db: sqlite3.Connection) -> None` that creates:
     ```
     CREATE TABLE IF NOT EXISTS state_history (
         id                INTEGER PRIMARY KEY AUTOINCREMENT,
         norad_id          INTEGER NOT NULL,
         epoch_utc         TEXT    NOT NULL,
         x_km              REAL    NOT NULL,
         y_km              REAL    NOT NULL,
         z_km              REAL    NOT NULL,
         vx_km_s           REAL    NOT NULL,
         vy_km_s           REAL    NOT NULL,
         vz_km_s           REAL    NOT NULL,
         cov_x_km2         REAL    NOT NULL,
         cov_y_km2         REAL    NOT NULL,
         cov_z_km2         REAL    NOT NULL,
         nis               REAL    NOT NULL,
         confidence        REAL    NOT NULL,
         anomaly_type      TEXT,
         message_type      TEXT    NOT NULL
     )
     ```
     Add index: `CREATE INDEX IF NOT EXISTS idx_state_history_norad_epoch ON state_history (norad_id, epoch_utc)`
   - Why: F-041 requires serving time-series history per object. In-memory filter state only holds the latest snapshot plus a short NIS window. Persistent storage is needed.
   - Dependencies: none
   - Risk: Low

2. **Define module-level application state dataclass / typed dict** (`backend/main.py`)
   - Action: Define a module-level mutable namespace (a simple class or module-level dict stored on `app.state`) to hold:
     - `db: sqlite3.Connection` -- shared database connection
     - `catalog_entries: list[dict]` -- loaded from `ingest.load_catalog_config()`
     - `filter_states: dict[int, dict]` -- keyed by NORAD ID, values are `kalman.init_filter()` return dicts
     - `event_bus: asyncio.Queue` -- the in-process event bus
     - `ws_connections: set[WebSocket]` -- active WebSocket connections
     - `background_tasks: list[asyncio.Task]` -- for cancellation on shutdown
   - Why: All endpoints and background tasks need shared access to these objects. Using `app.state` is the standard FastAPI pattern for lifespan-scoped state.
   - Dependencies: none
   - Risk: Low

### Phase 2: WebSocket connection manager

3. **Implement `ConnectionManager` class** (`backend/main.py`)
   - Action: Add a class `ConnectionManager` with methods:
     - `async def connect(self, websocket: WebSocket) -> None` -- accept and add to internal `set[WebSocket]`
     - `async def disconnect(self, websocket: WebSocket) -> None` -- remove from set
     - `async def broadcast(self, message: dict) -> None` -- `json.dumps` the message and call `websocket.send_text()` for each connection; catch and log exceptions per-connection (do not let one broken client block others); remove failed connections
     - `def active_count(self) -> int` -- return len of connections set
   - Why: F-044 requires supporting 5+ simultaneous connections. A dedicated manager encapsulates connection lifecycle and ensures broadcast failures on one client do not affect others.
   - Dependencies: none
   - Risk: Low

4. **Instantiate `ConnectionManager` at module level** (`backend/main.py`)
   - Action: Create a single `ConnectionManager` instance: `ws_manager = ConnectionManager()`
   - Why: Must be accessible to both the WebSocket endpoint handler and the background broadcast task.
   - Dependencies: step 3
   - Risk: Low

### Phase 3: Lifespan startup and shutdown

5. **Implement lifespan startup** (`backend/main.py`)
   - Action: Replace the `raise NotImplementedError` in the startup section of the `lifespan` context manager with:
     1. Read `NBODY_DB_PATH` env var (default `data/catalog/tle_cache.db`)
     2. Read `NBODY_CATALOG_CONFIG` env var (default `data/catalog/catalog.json`)
     3. Call `ingest.init_catalog_db(db_path)` to get the `sqlite3.Connection`
     4. Call `_ensure_state_history_table(db)` (from Phase 1 step 1)
     5. Call `anomaly.ensure_alerts_table(db)` (depends on anomaly.py plan being implemented; if not yet available, guard with a try/except and log a warning -- do NOT skip table creation silently)
     6. Call `ingest.load_catalog_config(catalog_config_path)` to get `catalog_entries`
     7. Create `asyncio.Queue()` as `event_bus`
     8. Store all of the above on `app.state`: `app.state.db`, `app.state.catalog_entries`, `app.state.filter_states = {}`, `app.state.event_bus`
     9. Create and store background tasks:
        - `asyncio.create_task(_ingest_loop_task(app))` -- wraps `ingest.run_ingest_loop`
        - `asyncio.create_task(_processing_loop_task(app))` -- event consumer
     10. Store tasks in `app.state.background_tasks` for shutdown cancellation
   - Why: The lifespan context manager is the correct place for async resource initialization in modern FastAPI. Background tasks must be created here so they run for the lifetime of the application.
   - Dependencies: steps 1, 2
   - Risk: Medium -- startup failure (missing config file, bad DB path) must produce a clear error message, not a silent hang. Add structured logging for each step.

6. **Implement lifespan shutdown** (`backend/main.py`)
   - Action: Replace the `raise NotImplementedError` in the shutdown section (after `yield`) with:
     1. Cancel all tasks in `app.state.background_tasks` and await their cancellation
     2. Close `app.state.db`
     3. Log shutdown complete
   - Why: Clean resource release. Prevents SQLite lock files and orphaned asyncio tasks.
   - Dependencies: step 5
   - Risk: Low

### Phase 4: Background tasks (the core event loop)

7. **Implement `_ingest_loop_task`** (`backend/main.py`)
   - Action: Add an `async def _ingest_loop_task(app: FastAPI) -> None` function that:
     1. Wraps `ingest.run_ingest_loop(db_path, catalog_config_path, event_bus)` in a try/except
     2. On `asyncio.CancelledError`: log and return cleanly
     3. On any other exception: log the error, sleep 60 seconds, and restart the loop (resilience per NF-010)
   - Why: The ingest loop must run continuously as a background asyncio task. It communicates with the processing loop solely through the event bus queue.
   - Dependencies: step 5
   - Risk: Low

8. **Implement `_processing_loop_task`** (`backend/main.py`)
   - Action: Add an `async def _processing_loop_task(app: FastAPI) -> None` function that:
     1. Loops forever: `event = await app.state.event_bus.get()`
     2. When event type is `catalog_update`:
        - For each entry in `app.state.catalog_entries`:
          - Call `ingest.get_latest_tle(app.state.db, norad_id)` to get the newest TLE
          - If no TLE exists, skip (log warning)
          - If no filter exists for this NORAD ID in `app.state.filter_states`:
            - Call `propagator.tle_to_state_vector_eci_km(tle_line1, tle_line2, epoch_utc)` to get initial state
            - Call `kalman.init_filter(state, epoch_utc, process_noise_q=kalman.OBJECT_CLASS_Q[entry["object_class"]])` to create filter
            - Store in `app.state.filter_states[norad_id]`
            - Build a `state_update` message and broadcast
          - If filter exists:
            - Call `kalman.predict(filter_state, epoch_utc, tle_line1, tle_line2)`
            - Call `propagator.tle_to_state_vector_eci_km(...)` to get the observation
            - Call `kalman.update(filter_state, observation, epoch_utc)`
            - Call `anomaly.classify_anomaly(norad_id, nis_history, innovation, is_active_satellite)` where `is_active_satellite = (entry["object_class"] == "active_satellite")`
            - If anomaly detected:
              - Call `anomaly.record_anomaly(db, norad_id, epoch_utc, anomaly_type, nis)`
              - Call `anomaly.trigger_recalibration(norad_id, anomaly_type, epoch_utc)`
              - Call `kalman.recalibrate(filter_state, observation, epoch_utc)` and replace filter state
              - Build and broadcast an `anomaly` message
              - Build and broadcast a `recalibration` message
            - Else: build and broadcast a `state_update` message
          - Write state snapshot row to `state_history` table
     3. On `asyncio.CancelledError`: log and return
     4. On any per-object exception: log error, skip that object, continue to next (do not crash the loop)
   - Why: This is the central orchestration point. It connects the ingest event bus output to the kalman/anomaly pipeline and the WebSocket broadcast. Running it as a separate task from the ingest loop keeps the concerns separated and prevents a slow filter update from blocking the next poll.
   - Dependencies: steps 1, 5, 7
   - Risk: **High** -- This function has the most logic and the most integration points. Errors here can cause silent data loss or stale WebSocket streams. Mitigations: (a) per-object try/except so one bad object does not block others; (b) structured logging at each stage; (c) integration test (see test strategy).

9. **Implement `_build_ws_message` helper** (`backend/main.py`)
   - Action: Add a function `def _build_ws_message(norad_id: int, filter_state: dict, message_type: str, anomaly_type: Optional[str] = None) -> dict` that constructs the architecture Section 3.5 message:
     ```python
     {
         "type": message_type,                      # "state_update" | "anomaly" | "recalibration"
         "norad_id": norad_id,
         "epoch_utc": last_epoch_utc.isoformat(),   # ISO-8601 with Z suffix
         "eci_km": [x, y, z],                       # from filter state_eci_km[:3]
         "eci_km_s": [vx, vy, vz],                  # from filter state_eci_km[3:]
         "covariance_diagonal_km2": [P[0,0], P[1,1], P[2,2]],  # position covariance only
         "nis": nis,
         "confidence": confidence,
         "anomaly_type": anomaly_type               # null for state_update
     }
     ```
     Numpy arrays must be converted to plain Python lists (`.tolist()`) for JSON serialization. The `epoch_utc` field must always end with `Z` suffix (UTC).
   - Why: F-043 requires strict conformance to the schema. Centralizing message construction prevents schema drift across the multiple broadcast sites in step 8.
   - Dependencies: none
   - Risk: Low

## Risks and mitigations

- **Risk:** SQLite concurrent access from async background tasks and sync REST handlers. SQLite with WAL mode supports one writer + multiple readers, but the `sqlite3` module is not async-safe. -- **Mitigation:** All DB writes happen in the single `_processing_loop_task` (serialized). REST reads are non-blocking with WAL. If contention appears under load, wrap DB access in `asyncio.to_thread()` (post-POC optimization, not needed for 5-client demo).

- **Risk:** `_processing_loop_task` processes all catalog objects sequentially on each event. For 50 objects, if each Kalman update takes 100ms (NF-001 ceiling), that is 5 seconds total before the next event is consumed. -- **Mitigation:** Acceptable for POC at 30-minute polling intervals. The 5-second batch is well within NF-002's 500ms per-object broadcast target (broadcast happens per-object within the loop, not after the entire batch). Log timing to validate.

- **Risk:** WebSocket broadcast to a slow client blocks the broadcast loop. -- **Mitigation:** `ConnectionManager.broadcast()` catches per-client exceptions and removes failed connections. Consider adding a send timeout (e.g., 5 seconds) per client in the broadcast loop.

- **Risk:** Startup fails if `anomaly.ensure_alerts_table()` is not yet implemented (anomaly.py plan may not be executed yet). -- **Mitigation:** Guard with try/except; log a warning. The alerts table is needed for `record_anomaly()` calls in the processing loop, which are themselves guarded by anomaly detection logic. If anomaly.py stubs still raise `NotImplementedError`, the processing loop's per-object try/except catches it and logs without crashing.

- **Risk:** `_processing_loop_task` calls `anomaly.classify_anomaly()` which currently raises `NotImplementedError`. -- **Mitigation:** The per-object try/except in step 8 handles this. When the anomaly plan is implemented, the integration will work without changes to main.py. Log the `NotImplementedError` at DEBUG level so it does not spam during development.

## Conflict: architecture.md anomaly_type values vs. requirements/main.py docstring

> **Conflict:** The architecture document Section 3.5 specifies `anomaly_type` values as `"maneuver" | "drag" | "conjunction"`, but the main.py module docstring (line 19) and `anomaly.py` constants use `"maneuver" | "drag_anomaly" | "filter_divergence"`. The requirements document F-031 specifies the classification types as `maneuver`, `drag_anomaly`, `filter_divergence`. Resolution: the **requirements document and anomaly.py constants are authoritative**. The architecture.md Section 3.5 message format example should be updated to use `"drag_anomaly"` instead of `"drag"` and `"filter_divergence"` instead of `"conjunction"`. This deviation must be noted in the implementation and architecture.md should be updated. Flagged for human decision before implementation.
