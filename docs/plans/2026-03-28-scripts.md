# Implementation Plan: Demo and Replay Scripts (F-060 through F-063)
Date: 2026-03-28
Status: Approved

## Summary

Implement `scripts/replay.py` and `scripts/seed_maneuver.py` to support the funding demo. The replay script loads pre-cached TLEs from the SQLite database and drives them through the full ingest-to-kalman-to-anomaly processing pipeline, writing state history and anomaly records directly to the database without requiring a running FastAPI server or live Space-Track connectivity. The maneuver injection script computes a post-maneuver ECI state from a delta-V specification in the RSW (radial/along-track/cross-track) frame, converts it to a synthetic TLE via the SGP4 state-to-elements reverse path, and inserts it into the TLE cache so the next processing cycle (either replay or live server) detects the anomaly. Both scripts are single-command CLI tools that satisfy F-063.

## Requirements addressed

- **F-060 [DEMO]**: Replay script simulates observe-predict-validate loop using pre-cached historical TLEs, without live Space-Track connectivity.
- **F-061 [DEMO]**: Maneuver injection script introduces a delta-V event into a selected object's state, causing a detectable anomaly through the filter and visualization.
- **F-062**: Maneuver injection script accepts: NORAD ID, delta-V magnitude (m/s), direction (along-track / cross-track / radial), and epoch offset from current time.
- **F-063**: Both scripts runnable from a single terminal command.
- **NF-020**: Demo launchable from single shell script (scripts must not require interactive prompts).
- **NF-023**: Anomaly injection produces visible browser response within 10 seconds (informs design choice for seed_maneuver to insert into DB so the running server picks it up on the next processing cycle).
- **TD-023**: Resolves the P1 tech debt item for unimplemented scripts.

## Design decisions

### Decision 1: replay.py mechanism -- Direct backend module import (not WebSocket client)

**Chosen approach:** replay.py directly imports and calls backend modules (`ingest`, `propagator`, `kalman`, `anomaly`) synchronously, writing results to the SQLite database. It does NOT require the FastAPI server to be running. When the server subsequently starts, it reads pre-populated state_history and alerts tables.

**Rejected alternative:** WebSocket client that connects to a running server and feeds TLEs in accelerated time. This was rejected because:
1. It requires the server to already be running, adding a startup ordering dependency that conflicts with F-063 (single command) and NF-020 (no interactive prompts).
2. The server's ingest loop would conflict with replay data -- the ingest loop polls Space-Track on its own schedule, and there is no API endpoint to inject TLEs.
3. Demo stability is higher when the script writes directly to the database: no network calls, no server state management, no WebSocket connection failures.

**Implication:** replay.py reuses `_process_single_object`-equivalent logic from `main.py`. To avoid code duplication, this plan extracts the core processing logic into a shared function in a new module `backend/processing.py`. The function is called by both `main.py._process_single_object` and `replay.py`.

### Decision 2: seed_maneuver.py mechanism -- Synthetic TLE insertion into SQLite cache

**Chosen approach (option b from the prompt):** Compute a post-maneuver ECI state vector by applying the delta-V in the RSW frame (converted to ECI), then generate a synthetic TLE from the perturbed state using `sgp4`'s reverse initialization (`Satrec.sgp4init` or element fitting), and insert it into the `tle_catalog` SQLite table with the maneuver epoch. The running server's processing loop (or replay.py) then picks up this TLE on the next cycle and the Kalman filter detects the anomaly.

**Why this approach works:**
- It does not require the server to be running at injection time (demo setup flexibility).
- It does not require modifying in-memory filter state (which is inaccessible from an external script).
- The existing `_process_single_object` pipeline in main.py already handles new TLEs from the cache -- no server-side code changes needed.
- The synthetic TLE, when propagated by the server's filter, will produce a residual because the filter's prior state is based on the pre-maneuver orbit.

**RSW-to-ECI conversion approach:** The RSW (Radial-Along-Cross) frame is defined relative to the object's current ECI position and velocity:
- Radial (R): unit vector along position: `r_hat = r / |r|`
- Along-track (S): approximately along velocity; computed as the component of velocity perpendicular to R then normalized. More precisely: `s_hat = (v - (v . r_hat) * r_hat) / |v - (v . r_hat) * r_hat|`
- Cross-track (W): completes the right-hand triad: `w_hat = r_hat x s_hat`

The delta-V in RSW is converted to ECI by: `delta_v_eci = dv_r * r_hat + dv_s * s_hat + dv_w * w_hat`.

**Synthetic TLE generation:** The sgp4 library's `Satrec` can be initialized from orbital elements via `sgp4init`. The plan uses the following approach:
1. Propagate the pre-maneuver TLE to the maneuver epoch to get the base ECI state.
2. Apply delta-V to the velocity components.
3. Convert the perturbed ECI state to Keplerian orbital elements (a, e, i, RAAN, argp, M).
4. Use these elements to construct TLE line strings with the `sgp4` library.

This requires a helper function `eci_state_to_tle_lines()` that converts a 6-element ECI state vector back to TLE format. This is the inverse of what `propagator.py` does. The function will be placed in `scripts/seed_maneuver.py` itself (not in `propagator.py`) because:
- It is a demo-only utility, not part of the core propagation engine.
- The inverse mapping (ECI -> TLE) is inherently lossy (SGP4 is not a simple Keplerian propagator), so the generated TLE will have small errors compared to the true perturbed state. This is acceptable for demo purposes because the errors are smaller than the delta-V perturbation.

**Rejected alternatives:**
- (a) Modify filter state in SQLite: filter state is in-memory only (in `app.state.filter_states` dict in `main.py`), not persisted to SQLite. Not feasible.
- (c) Write a "maneuver event" record to a special table: requires modifying `main.py` processing loop to read this table, adding new server-side code. More complex for no benefit over synthetic TLE insertion.
- (d) Call a dedicated HTTP endpoint: requires server to be running. No such endpoint exists. Would need new server code.

### Decision 3: Shared processing logic extraction

To avoid duplicating the predict-update-anomaly-broadcast pipeline between `main.py._process_single_object` and `replay.py`, the core processing logic will be extracted into `backend/processing.py`. This module provides a synchronous function `process_single_object()` that:
- Takes a DB connection, TLE record, catalog entry, and filter_states dict.
- Runs the predict -> update -> anomaly detect -> recalibrate pipeline.
- Writes state_history and alerts rows to the database.
- Returns a list of WebSocket message dicts (for the caller to broadcast or discard).

`main.py._process_single_object` becomes a thin wrapper that calls `processing.process_single_object()` and broadcasts the returned messages. `replay.py` calls it directly and discards the WebSocket messages (or optionally prints them to stdout for debugging).

> **Conflict:** This plan proposes extracting logic from `main.py._process_single_object` into a new module `backend/processing.py`. This is a refactor of existing code in `main.py` that the implementer agent's rules say "does not refactor surrounding code unless the plan explicitly calls for it." This plan explicitly calls for it. The alternative -- duplicating ~100 lines of processing logic in `replay.py` -- would violate DRY and create a maintenance burden where bugs fixed in one location would need to be fixed in both. The refactor is scoped narrowly: extract, do not redesign.

### Phase 3: Implement seed_maneuver.py

5. **Implement RSW-to-ECI conversion helper** (`scripts/seed_maneuver.py`)
   - Action: Add a function `rsw_to_eci_delta_v_km_s()`:
     ```
     def rsw_to_eci_delta_v_km_s(
         position_eci_km: NDArray[np.float64],
         velocity_eci_km_s: NDArray[np.float64],
         delta_v_radial_km_s: float,
         delta_v_along_track_km_s: float,
         delta_v_cross_track_km_s: float,
     ) -> NDArray[np.float64]:
     ```
     Computes the RSW frame unit vectors from position and velocity:
     - `r_hat = position / |position|`
     - `w_hat = (position x velocity) / |position x velocity|`
     - `s_hat = w_hat x r_hat`
     Returns `delta_v_radial * r_hat + delta_v_along_track * s_hat + delta_v_cross_track * w_hat`.
     All inputs and outputs in km/s (ECI J2000). The caller converts m/s to km/s before calling.
   - Why: F-062 requires direction specification in RSW frame. Domain rule: all internal state vectors in ECI J2000 -- the RSW frame is used only for the conversion, not stored.
   - Dependencies: None (uses numpy only).
   - Risk: Low -- standard orbital mechanics frame rotation.

6. **Implement ECI-state-to-Keplerian-elements helper** (`scripts/seed_maneuver.py`)
   - Action: Add a function `eci_to_keplerian()`:
     ```
     def eci_to_keplerian(
         position_eci_km: NDArray[np.float64],
         velocity_eci_km_s: NDArray[np.float64],
         mu_km3_s2: float = 398600.4418,
     ) -> dict:
     ```
     Converts ECI state to classical Keplerian elements: semi-major axis (km), eccentricity, inclination (rad), RAAN (rad), argument of perigee (rad), true anomaly (rad). Uses standard orbital mechanics formulas (vis-viva, angular momentum vector, eccentricity vector, node vector). Returns a dict with keys: `a_km`, `e`, `i_rad`, `raan_rad`, `argp_rad`, `true_anomaly_rad`.
   - Why: Needed to construct a synthetic TLE from the perturbed ECI state.
   - Dependencies: None.
   - Risk: Low -- well-known formulas, but edge cases (circular/equatorial orbits) need guarding. For POC LEO objects (ISS, Starlink), these edge cases are unlikely but should use small-epsilon fallbacks.

7. **Implement Keplerian-to-TLE-lines helper** (`scripts/seed_maneuver.py`)
   - Action: Add a function `keplerian_to_tle_lines()`:
     ```
     def keplerian_to_tle_lines(
         norad_id: int,
         epoch_utc: datetime.datetime,
         a_km: float,
         e: float,
         i_rad: float,
         raan_rad: float,
         argp_rad: float,
         mean_anomaly_rad: float,
         bstar: float,
     ) -> tuple[str, str]:
     ```
     Constructs TLE line 1 and line 2 strings from the given orbital elements. Computes mean motion from semi-major axis via `n = sqrt(mu / a^3)` in rev/day. Formats the TLE epoch field from the UTC datetime. Sets the B* drag term from the original TLE's B* value (passed through as `bstar`). Computes TLE checksums using `ingest._tle_checksum` (or a local reimplementation).

     **True anomaly to mean anomaly conversion:** The Keplerian elements from step 6 produce true anomaly. TLEs require mean anomaly. The conversion is: eccentric anomaly `E = 2 * atan2(sqrt(1-e) * sin(nu/2), sqrt(1+e) * cos(nu/2))`, then mean anomaly `M = E - e * sin(E)`.
   - Why: Needed to produce a valid TLE that SGP4 can parse and propagate.
   - Dependencies: Requires step 6.
   - Risk: Medium -- TLE formatting is fiddly (fixed-width fields, checksum computation, epoch encoding). Must produce TLEs that pass `ingest.validate_tle()`. Test thoroughly.

8. **Implement `inject_maneuver()` function** (`scripts/seed_maneuver.py`)
   - Action: Replace the `NotImplementedError` stub with the following logic:
     a. Resolve `db_path` from `--db` argument (default: `NBODY_DB_PATH` env var, fallback `data/catalog/tle_cache.db`).
     b. Open DB via `ingest.init_catalog_db(db_path)`.
     c. Retrieve the latest TLE for the given NORAD ID via `ingest.get_latest_tle(db, norad_id)`. If no TLE found, print error and exit with code 1.
     d. Compute maneuver epoch: `maneuver_epoch_utc = datetime.datetime.now(datetime.timezone.utc) + timedelta(seconds=epoch_offset_s)`.
     e. Propagate the pre-maneuver TLE to the maneuver epoch via `propagator.tle_to_state_vector_eci_km(tle_line1, tle_line2, maneuver_epoch_utc)` to get the base ECI state.
     f. Convert `delta_v_m_s` to km/s: `delta_v_km_s = delta_v_m_s / 1000.0`. Domain rule: F-062 specifies m/s input, internal units are km/s.
     g. Map the `direction` argument to RSW components:
        - `"along-track"` -> `(0, delta_v_km_s, 0)`
        - `"cross-track"` -> `(0, 0, delta_v_km_s)`
        - `"radial"` -> `(delta_v_km_s, 0, 0)`
     h. Call `rsw_to_eci_delta_v_km_s()` to get the ECI delta-V vector.
     i. Add delta-V to the velocity components of the base state: `perturbed_velocity = base_velocity + delta_v_eci`.
     j. Convert the perturbed ECI state (base position + perturbed velocity) to Keplerian elements via `eci_to_keplerian()`.
     k. Convert true anomaly to mean anomaly (within `keplerian_to_tle_lines` or as a helper).
     l. Extract B* from the original TLE (parse from TLE line 1 field, columns 53-61).
     m. Generate synthetic TLE lines via `keplerian_to_tle_lines()`.
     n. Validate the synthetic TLE via `ingest.validate_tle()`. If validation fails, print error and exit.
     o. Insert the synthetic TLE into the cache via `ingest.cache_tles()` with `fetched_at_utc = now_utc`.
     p. Print confirmation: `Injected maneuver for NORAD {norad_id}: delta-V={delta_v_m_s} m/s {direction} at epoch {maneuver_epoch_utc}`.
   - Why: Satisfies F-061 and F-062.
   - Dependencies: Requires steps 5, 6, 7.
   - Risk: Medium -- the synthetic TLE will have small errors relative to the true perturbed orbit because SGP4 element sets are not exact representations of arbitrary ECI states. For demo purposes, the delta-V perturbation (0.5 m/s default) is orders of magnitude larger than the TLE fitting error, so the anomaly will be clearly detectable.

## Risks and mitigations

- **Risk:** Synthetic TLE fitting error. Converting an arbitrary ECI state to TLE via Keplerian elements and then propagating with SGP4 introduces fitting error because SGP4 includes secular and periodic perturbation corrections that a pure Keplerian conversion does not account for. -- **Mitigation:** For demo delta-V magnitudes (0.5-5 m/s), the TLE fitting error (~100m position) is 1-2 orders of magnitude smaller than the maneuver signature. The filter will detect the anomaly clearly. Document this limitation in the script's docstring.

- **Risk:** Refactoring `main.py._process_single_object` into `processing.py` could introduce a regression. -- **Mitigation:** The extraction is purely structural (move code, adjust imports). Existing tests for main.py must pass after the refactor. Run the full test suite immediately after step 2.

- **Risk:** TLE line formatting in `keplerian_to_tle_lines()` is error-prone (fixed-width fields, exponential notation for B*). -- **Mitigation:** Validate every generated TLE through `ingest.validate_tle()` before insertion. Add a round-trip unit test that generates a TLE and verifies it can be parsed and propagated by SGP4.

- **Risk:** `replay.py` processing order. If TLEs for multiple objects have the same epoch, processing order could affect filter states. -- **Mitigation:** Sort by (epoch_utc, norad_id) for deterministic ordering. This matches the server's behavior (processes all objects per catalog_update event).
