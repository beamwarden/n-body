# Implementation Plan: Browser Visualization Frontend (F-050 through F-056)
Date: 2026-03-28
Status: Approved

## Summary

Implement the complete browser-based visualization frontend for the ne-body SSA platform. This plan covers: establishing a WebSocket connection and message routing (`main.js`), rendering a 3D CesiumJS globe with satellite positions, confidence color-coding, and uncertainty ellipsoids (`globe.js`), building D3.js residual and NIS time-series charts (`residuals.js`), implementing an anomaly alert feed (`alerts.js`), and wiring globe click selection to update all per-object panels. These are all `[DEMO]`-tagged requirements critical to the funding presentation.

## Requirements addressed

- **F-050 [DEMO]**: 3D globe with real-time satellite position updates from WebSocket
- **F-051 [DEMO]**: Confidence-based color coding (green > 0.85, amber 0.60-0.85, red < 0.60)
- **F-052 [DEMO]**: 3-sigma uncertainty ellipsoids from covariance diagonal
- **F-053 [DEMO]**: Anomaly visual highlight within 2 seconds
- **F-054 [DEMO]**: Residual magnitude and NIS time-series chart with +/-2-sigma noise band
- **F-055 [DEMO]**: Anomaly alert feed with object name, time, type, resolution status
- **F-056**: Click-to-select on globe updates residual chart and per-object panels
- **NF-003**: 30 FPS on integrated GPU with 50 objects (governs entity strategy choice)
- **NF-012**: Reconnecting WebSocket client receives full current state within 5 seconds
- **NF-022**: Legible text at 1920x1080 on projected display at 3 meters
- **NF-040**: No credentials in source (Cesium Ion token via backend `/config` endpoint)
- **TD-018**: Resolve CesiumJS Ion token hardcoding

## Files affected

- `frontend/index.html` -- Add Cesium Ion token injection via `/config` fetch; add `<noscript>` fallback; adjust layout for demo legibility (NF-022)
- `frontend/src/main.js` -- Implement WebSocket connection, reconnection logic, message routing, initial catalog fetch, module initialization, shared event bus for object selection
- `frontend/src/globe.js` -- Implement CesiumJS viewer initialization, ECI-to-ECEF conversion, satellite entity management, confidence color mapping, uncertainty ellipsoid rendering, anomaly highlight animation, click selection handler
- `frontend/src/residuals.js` -- Implement D3.js dual-panel chart (residual magnitude + NIS), per-object data store, +/-2-sigma noise band, NIS threshold line, incremental append on WebSocket messages, object switching
- `frontend/src/alerts.js` -- Implement scrolling alert feed DOM, anomaly event rendering, status lifecycle (active -> recalibrating -> resolved), time-to-resolution display
- `backend/main.py` -- Add `GET /config` endpoint returning `{ cesium_ion_token: <from env> }` (resolves TD-018)

### ECI-to-ECEF conversion (blocking design decision)

**Decision: frontend performs ECI-to-ECEF rotation in `globe.js` using GMST.**

Rationale:
- The backend WebSocket schema (architecture section 3.5) sends `eci_km` only. Adding geodetic fields would require a schema change affecting the architecture document, all backend broadcast code, and all consumers. This is disproportionate for a POC.
- The conversion is a single 3x3 rotation matrix applied to the position vector. The rotation angle is GMST (Greenwich Mean Sidereal Time), computable from the message's `epoch_utc` timestamp using a standard formula (IAU 1982 or simplified Vallado formula).
- CesiumJS expects ECEF Cartesian3 in meters. The conversion is: `R_z(-GMST) * eci_km * 1000 = ecef_meters`.
- No external JS library is needed. The GMST formula is ~10 lines of code.
- The architecture document section 3.2 states: "conversion is the responsibility of the API layer." The browser is at the API boundary, so performing the conversion there is consistent. The architecture document section 3.6 does not prescribe where the conversion happens.

**GMST formula** (Vallado, simplified for POC accuracy):

```
JD_ut1 = julian_date(epoch_utc)
T_ut1 = (JD_ut1 - 2451545.0) / 36525.0
gmst_sec = 67310.54841 + (876600h + 8640184.812866)*T + 0.093104*T^2 - 6.2e-6*T^3
gmst_rad = (gmst_sec % 86400) / 86400 * 2 * pi
```

This gives GMST to sub-arcsecond accuracy, more than sufficient for the ~1 km positional uncertainty of TLE-derived states.

**Frame safety note:** The ECI vectors from the backend are technically GCRS (per TD-003), which differs from true J2000 by ~20 milliarcseconds. The GMST rotation converts GCRS to ITRS (ECEF). The error introduced by treating GCRS as J2000 for GMST purposes is sub-meter, negligible for POC visualization.

### CZML vs. Entity API (design decision)

**Decision: use the CesiumJS Entity API directly, not CZML DataSource.**

Rationale:
- The architecture document section 3.6.1 says "CZML data source that is patched with each incoming WebSocket state update." However, CZML patching requires serializing updates into CZML packet format and then calling `CzmlDataSource.process()`, which is an unnecessary serialization/deserialization round-trip when we already have the data in JavaScript objects.
- The Entity API allows direct property updates (`entity.position = new Cartesian3(...)`) which is simpler, faster, and easier to debug.
- For 50 objects at 30-minute update intervals, performance is not a concern with either approach.
- CesiumJS Entity API supports all required features: billboards, ellipsoids, color properties, and click selection via `viewer.selectedEntityChanged`.

> **Deviation from architecture section 3.6.1:** This plan uses the Entity API instead of CZML DataSource. Justification: simpler code, no serialization overhead, identical visual result, and the CZML approach adds complexity without benefit at POC scale (50 objects, 30-minute updates). If post-POC scaling to 10k+ objects requires batch updates, CZML or 3D Tiles would be reconsidered.

### D3 chart update strategy (design decision)

**Decision: incremental append with a sliding window, not full redraw.**

Rationale:
- Each WebSocket message for the selected object appends one data point. Full SVG redraw on every message is wasteful.
- Maintain a fixed-length data buffer per object (e.g., last 100 data points, covering ~50 hours at 30-minute intervals). When a new point arrives, push to the buffer, shift the oldest if at capacity, then update the D3 scales and re-bindthe line/area paths.
- The `+/-2-sigma` noise band and NIS threshold line are static reference elements redrawn only when the selected object changes.
- On object switch (`selectObject`), clear the SVG content and redraw from the target object's full buffer.

### Inter-module communication for selection (design decision)

**Decision: callback-based, coordinated by `main.js`. No separate event bus library.**

Rationale:
- `main.js` already imports all three modules. It passes an `onSelect` callback to `globe.js`'s `setupSelectionHandler`. When globe fires the callback, `main.js` calls `residuals.selectObject()` and optionally highlights relevant alerts.
- This avoids introducing a pub/sub library or custom EventTarget for a 3-module application.
- If post-POC the frontend grows, a `CustomEvent` on `document` or a minimal event emitter can replace the callbacks.

## Implementation steps

### Phase 1: Backend config endpoint and index.html token injection

1. **Add `GET /config` endpoint** (`backend/main.py`)
   - Action: Add a new route `@app.get("/config")` that returns `{"cesium_ion_token": os.environ.get("CESIUM_ION_TOKEN", "")}`. Import `os` at module top.
   - Why: Resolves TD-018. The Cesium Ion token must not be hardcoded in frontend source. The backend reads it from the environment and serves it to the frontend at runtime.
   - Dependencies: none
   - Risk: Low. If `CESIUM_ION_TOKEN` is unset, the empty string causes CesiumJS to fail visibly (no imagery tiles), which is an obvious configuration error rather than a silent one.

2. **Update `index.html` for token fetch and layout** (`frontend/index.html`)
   - Action:
     - Remove the comment block about token injection (lines 15-24) and replace with a brief comment referencing `/config`.
     - Add an inline `<script>` before the module script tag that fetches `/config` and sets `window.CESIUM_ION_TOKEN`. The module script tag must wait for this. Two approaches: (a) use a top-level `await` in `main.js` to fetch config before calling `initGlobe`, or (b) use a synchronous blocking fetch in a classic script. Approach (a) is cleaner and ES2022-compliant. Use approach (a): the module `main.js` will fetch `/config` as its first action.
     - Adjust CSS for demo legibility (NF-022): increase `#header` font-size to 18px, ensure `#side-panel` text is at least 14px, add explicit `font-size` to `#alert-panel` and `#residual-chart`.
   - Why: Provides runtime token injection and ensures demo readability on projected displays.
   - Dependencies: step 1
   - Risk: Low

### Phase 2: WebSocket connection and message routing (`main.js`)

3. **Implement `connectWebSocket`** (`frontend/src/main.js`)
   - Action: Create a WebSocket to the given URL. Attach `onopen`, `onclose`, `onerror`, `onmessage` handlers. On `onmessage`, parse JSON and call `routeMessage`. On `onclose`/`onerror`, implement reconnection with exponential backoff (initial 1s, max 30s, with jitter). Log connection state changes to `console.info`.
   - Why: F-050 requires real-time updates from WebSocket. NF-012 requires reconnection with full state recovery.
   - Dependencies: none
   - Risk: Low

4. **Implement `routeMessage`** (`frontend/src/main.js`)
   - Action: Switch on `message.type`:
     - `"state_update"`: call `updateSatellitePosition(viewer, message)`, `updateUncertaintyEllipsoid(viewer, message.norad_id, message.covariance_diagonal_km2, message.eci_km)`. If `message.norad_id === selectedNoradId`, call `appendResidualDataPoint(chartState, message)`.
     - `"anomaly"`: call `highlightAnomaly(viewer, message.norad_id, message.anomaly_type)`, `addAlert(panelState, message)`. If `message.norad_id === selectedNoradId`, also call `appendResidualDataPoint(chartState, message)`.
     - `"recalibration"`: call `updateAlertStatus(panelState, message.norad_id, 'resolved', message.epoch_utc)`. Call `updateSatellitePosition(viewer, message)` and `updateUncertaintyEllipsoid(...)` since recalibration includes updated state.
   - Why: Central dispatch ensures each module receives only the messages it needs.
   - Dependencies: steps 5, 9, 13 (the modules being called). However, the routing logic can be written first with the understanding that called functions will be implemented in later steps.
   - Risk: Low

5. **Implement `fetchCatalog`** (`frontend/src/main.js`)
   - Action: `fetch(baseUrl + '/catalog')` then `.json()`. Return the array. Handle fetch errors with `console.error` and return empty array.
   - Why: On initial load and reconnection, the frontend needs the full catalog to initialize or re-sync globe entities.
   - Dependencies: none
   - Risk: Low

6. **Implement `initApp`** (`frontend/src/main.js`)
   - Action: This is the application entry point, called at module load via a top-level `initApp()` call at the bottom of `main.js`. Sequence:
     1. Fetch `/config` to get `cesium_ion_token`. Derive `backendBaseUrl` from `window.location` (same host, port 8000).
     2. Call `initGlobe('cesium-container', cesium_ion_token)` -- store returned viewer.
     3. Call `initResidualChart('residual-chart')` -- store returned chartState.
     4. Call `initAlertPanel('alert-panel')` -- store returned panelState.
     5. Call `setupSelectionHandler(viewer, onObjectSelected)` where `onObjectSelected` sets `selectedNoradId` and calls `residuals.selectObject(chartState, noradId)`.
     6. Call `fetchCatalog(backendBaseUrl)` to get initial object list. For each object, optionally place a placeholder entity on the globe (or wait for first WebSocket message).
     7. Call `connectWebSocket('ws://' + location.hostname + ':8000/ws/live')`.
   - Why: Orchestrates startup sequence. All module initialization must complete before WebSocket messages arrive.
   - Dependencies: steps 3, 4, 5 and all module init functions
   - Risk: Medium -- startup ordering matters. If WebSocket connects before globe is initialized, messages will fail. The sequence above ensures globe is ready first.

7. **Add module-level state variables** (`frontend/src/main.js`)
   - Action: Add module-scoped `let` variables: `let viewer = null; let chartState = null; let panelState = null; let selectedNoradId = null; let ws = null;`. These are shared across the exported functions within the module.
   - Why: Module state is needed to wire the callback from globe selection to residuals/alerts.
   - Dependencies: none
   - Risk: Low

### Phase 3: CesiumJS globe (`globe.js`)

8. **Implement ECI-to-ECEF conversion utility** (`frontend/src/globe.js`)
   - Action: Add a non-exported helper function `eciToEcefCartesian3(eci_km, epoch_utc_str)` that:
     1. Parses `epoch_utc_str` as a `Date` object.
     2. Computes Julian Date from the UTC Date.
     3. Computes GMST in radians using the Vallado formula (see Data Flow Changes section above).
     4. Applies the Z-rotation: `x_ecef = x_eci * cos(gmst) + y_eci * sin(gmst)`, `y_ecef = -x_eci * sin(gmst) + y_eci * cos(gmst)`, `z_ecef = z_eci`.
     5. Returns `new Cesium.Cartesian3(x_ecef * 1000, y_ecef * 1000, z_ecef * 1000)` (km to meters).
   - Why: CesiumJS uses ECEF Cartesian3 in meters. Backend sends ECI km. This is the only place the frame conversion happens on the frontend.
   - Dependencies: none
   - Risk: Medium -- GMST computation must be correct or all satellite positions will be rotated to wrong longitudes. Mitigation: validate against a known epoch (e.g., J2000.0 where GMST = 280.46 degrees) in unit tests.

   **Domain rule check:** This step introduces an ECI-to-ECEF frame conversion. The conversion is explicit, uses a documented formula, and is isolated in a single function. Variable names use `_km` and `_ecef` suffixes. Timestamps are UTC (`epoch_utc_str`). This is consistent with the architecture rule that conversions happen at the API boundary.

9. **Implement `initGlobe`** (`frontend/src/globe.js`)
   - Action:
     1. Set `Cesium.Ion.defaultAccessToken = ionToken`.
     2. Create `new Cesium.Viewer(containerId, { ... })` with options: `timeline: false, animation: false, baseLayerPicker: false, geocoder: false, homeButton: false, sceneModePicker: false, navigationHelpButton: false, infoBox: false, selectionIndicator: true`. These options remove UI clutter for the demo.
     3. Set initial camera view to show full Earth: `viewer.camera.flyTo({ destination: Cesium.Cartesian3.fromDegrees(0, 0, 35000000) })`.
     4. Store an internal `Map<norad_id, Cesium.Entity>` as a module-level variable `const entityMap = new Map()` for fast entity lookup.
     5. Return the viewer instance.
   - Why: Initializes the 3D globe (F-050). Minimal UI options keep focus on the data for the demo.
   - Dependencies: step 1 (token must be available)
   - Risk: Low. If Ion token is invalid, CesiumJS shows a grey globe without imagery -- visible but not a crash.

10. **Implement `confidenceColor`** (`frontend/src/globe.js`)
    - Action: Return `Cesium.Color.GREEN` if confidence > 0.85, `Cesium.Color.ORANGE` (amber) if confidence >= 0.60, `Cesium.Color.RED` otherwise.
    - Why: F-051 confidence color coding.
    - Dependencies: none
    - Risk: Low

11. **Implement `updateSatellitePosition`** (`frontend/src/globe.js`)
    - Action:
      1. Extract `norad_id`, `eci_km`, `epoch_utc`, `confidence` from `stateUpdate`.
      2. Call `eciToEcefCartesian3(eci_km, epoch_utc)` to get ECEF Cartesian3.
      3. Look up entity in `entityMap` by `norad_id`.
      4. If entity exists: update `entity.position = new Cesium.ConstantPositionProperty(cartesian3)`. Update `entity.billboard.color = new Cesium.ConstantProperty(confidenceColor(confidence))`.
      5. If entity does not exist: create a new `viewer.entities.add({ id: 'sat-' + norad_id, position: cartesian3, billboard: { image: <a circle/dot canvas data URL or Cesium pin>, color: confidenceColor(confidence), scale: 0.6 }, label: { text: String(norad_id), font: '14px monospace', fillColor: Cesium.Color.WHITE, style: Cesium.LabelStyle.FILL, verticalOrigin: Cesium.VerticalOrigin.BOTTOM, pixelOffset: new Cesium.Cartesian2(0, -12) }, properties: { norad_id: norad_id } })`. Add to `entityMap`.
    - Why: F-050 real-time position updates, F-051 color coding.
    - Dependencies: steps 8, 10
    - Risk: Low

12. **Implement `updateUncertaintyEllipsoid`** (`frontend/src/globe.js`)
    - Action:
      1. Compute semi-axes from covariance diagonal: `radii_km = [3 * sqrt(cov[0]), 3 * sqrt(cov[1]), 3 * sqrt(cov[2])]` (3-sigma).
      2. Convert to meters: `radii_m = radii_km.map(r => r * 1000)`.
      3. Compute ECEF position from `positionEciKm` using `eciToEcefCartesian3`.
      4. Look up ellipsoid entity in a separate `Map<norad_id, Cesium.Entity>` called `ellipsoidMap`.
      5. If exists: update `entity.position` and `entity.ellipsoid.radii = new Cesium.Cartesian3(radii_m[0], radii_m[1], radii_m[2])`.
      6. If not exists: create `viewer.entities.add({ id: 'ell-' + norad_id, position: cartesian3, ellipsoid: { radii: new Cesium.Cartesian3(...), material: Cesium.Color.WHITE.withAlpha(0.15), outline: true, outlineColor: Cesium.Color.CYAN.withAlpha(0.4) } })`. Add to `ellipsoidMap`.
      7. Clamp minimum radii to 5000 meters (5 km) to ensure ellipsoids are visible at globe zoom levels even when covariance is very small.
    - Why: F-052 uncertainty ellipsoid rendering at 3-sigma.
    - Dependencies: step 8
    - Risk: Medium -- The ellipsoid orientation is not rotated from ECI to ECEF. The covariance diagonal represents variances along ECI x/y/z axes, but the ellipsoid is rendered in ECEF. For a diagonal covariance (no cross-terms), this is an approximation: the ellipsoid shape is correct but its principal-axis orientation is rotated by the GMST angle relative to ground. For the POC this is acceptable because: (a) TLE-derived covariances are approximate to begin with, (b) the visual purpose is to show relative uncertainty magnitude, not precise orientation, and (c) a proper orientation would require rotating the full 3x3 covariance matrix, which the backend does not send (only the diagonal). Flag this as a known approximation.

    > **Known approximation:** Uncertainty ellipsoid axes are aligned with ECEF rather than the object's ECI frame. The shape (semi-axis lengths) is correct but the orientation relative to the orbit is not preserved. Correcting this requires the full covariance matrix and an ECI-to-ECEF rotation of the covariance tensor, which is out of scope for POC.

13. **Implement `highlightAnomaly`** (`frontend/src/globe.js`)
    - Action:
      1. Look up satellite entity in `entityMap` and ellipsoid entity in `ellipsoidMap`.
      2. Change billboard color to `Cesium.Color.YELLOW` (distinct from green/amber/red confidence colors).
      3. Increase billboard scale to `1.2` (double normal).
      4. Change ellipsoid material to `Cesium.Color.YELLOW.withAlpha(0.3)`.
      5. Set a `setTimeout` to revert to normal confidence-based color after 10 seconds (the highlight is a transient visual cue; the entity will be re-colored on the next `state_update` anyway).
    - Why: F-053 anomaly visual highlight. The 2-second latency requirement is met by the WebSocket delivery path (NF-002 specifies < 500ms) plus the immediate DOM update here.
    - Dependencies: steps 11, 12
    - Risk: Low

14. **Implement `setupSelectionHandler`** (`frontend/src/globe.js`)
    - Action:
      1. Register a `viewer.selectedEntityChanged` event listener.
      2. When a satellite entity is selected (entity ID starts with `'sat-'`), extract the NORAD ID from the entity's `properties.norad_id` value and call `onSelect(noradId)`.
      3. When selection is cleared (entity is `undefined`), call `onSelect(null)`.
    - Why: F-056 click-to-select.
    - Dependencies: step 9
    - Risk: Low

### Phase 4: D3.js residual chart (`residuals.js`)

15. **Implement per-object data store** (`frontend/src/residuals.js`)
    - Action: Add a module-level `const dataStore = new Map()` where keys are NORAD IDs and values are arrays of `{ epoch_utc: Date, residual_magnitude_km: number, nis: number, confidence: number, sigma2_km: number }`. Each array is capped at 100 entries (sliding window).
    - Why: Data must persist across messages so the chart can display history. The residual magnitude is computed as `Math.sqrt(eci_km[0]**2 + eci_km[1]**2 + eci_km[2]**2)` -- but this is the position vector, not the residual. See open question 1.
    - Dependencies: none
    - Risk: Low

    > **Ambiguity:** The WebSocket message schema (architecture section 3.5) sends `eci_km` (the current state estimate position) and `nis` (the normalized innovation squared). It does NOT send the raw residual vector. F-054 requires "residual magnitude." To display residual magnitude on the chart, the frontend needs either: (a) the backend to include a `residual_km` field in the WebSocket message, or (b) the frontend to compute a proxy. The NIS value is a normalized residual metric and can serve as the primary chart line. For "residual magnitude," the frontend can compute `sqrt(sum of covariance_diagonal_km2)` as a proxy for uncertainty magnitude, or the backend schema must be extended. See Open Questions.

16. **Implement `initResidualChart`** (`frontend/src/residuals.js`)
    - Action:
      1. Select the container div via `d3.select('#' + containerId)`.
      2. Create an SVG element sized to fill the container. Use a `ResizeObserver` to update dimensions on resize.
      3. Create two vertically stacked sub-charts within the SVG using `g` groups with `transform: translate()`:
         - **Top chart (60% height)**: Residual magnitude (or uncertainty proxy) vs. time. Y-axis: km. X-axis: UTC time.
         - **Bottom chart (40% height)**: NIS score vs. time. Y-axis: dimensionless. X-axis: UTC time (shared with top).
      4. Add axes, axis labels, grid lines. Use monospace font at 12px minimum for demo legibility (NF-022).
      5. Add a title text element showing "Object: [none selected]" initially.
      6. For the top chart: add an `<area>` element for the +/-2-sigma noise band (F-054), initially hidden.
      7. For the bottom chart: add a horizontal `<line>` for the NIS chi-squared threshold at 12.592 (chi-squared critical value, p=0.05, 6 dof), colored red, dashed.
      8. Return a `chartState` object: `{ svg, topGroup, bottomGroup, xScale, yScaleTop, yScaleBottom, lineResidual, lineNis, areaBand, selectedNoradId: null, containerEl }`.
    - Why: F-054 chart initialization. The dual-panel layout shows both residual behavior and filter health (NIS).
    - Dependencies: none
    - Risk: Low

17. **Implement `appendResidualDataPoint`** (`frontend/src/residuals.js`)
    - Action:
      1. Extract `norad_id`, `epoch_utc`, `nis`, `confidence`, `covariance_diagonal_km2` from `stateUpdate`.
      2. Compute uncertainty proxy: `sigma_total_km = Math.sqrt(covariance_diagonal_km2[0] + covariance_diagonal_km2[1] + covariance_diagonal_km2[2])`. This is the RSS of the 1-sigma position uncertainties.
      3. Build data point: `{ epoch_utc: new Date(epoch_utc), residual_magnitude_km: sigma_total_km, nis, confidence, sigma2_km: 2 * sigma_total_km }`.
      4. Push to `dataStore.get(norad_id)` (create array if missing). Enforce 100-entry cap.
      5. If `norad_id === chartState.selectedNoradId`, call internal `_redrawChart(chartState, norad_id)`.
    - Why: Incremental data accumulation per F-054.
    - Dependencies: steps 15, 16
    - Risk: Low

18. **Implement `_redrawChart` (internal)** (`frontend/src/residuals.js`)
    - Action:
      1. Get the data array for the selected NORAD ID from `dataStore`.
      2. Update `xScale` domain to `[oldest epoch, newest epoch]`.
      3. Update `yScaleTop` domain to `[0, max(residual_magnitude_km) * 1.2]`.
      4. Update `yScaleBottom` domain to `[0, max(nis, 15)]` (ensure threshold line is visible).
      5. Redraw axes with transitions (200ms for smooth updates).
      6. Re-bindthe top-chart line path to the residual magnitude data points. Use `d3.line()` with `curve(d3.curveMonotoneX)`.
      7. Update the +/-2-sigma noise band `<area>` using `d3.area()` where upper bound is `2 * sigma_total_km` and lower bound is `0` (residual magnitude is always positive, so the band is [0, 2-sigma]).
      8. Re-bind the bottom-chart line path to NIS data points.
      9. Add colored circles at each data point, colored by confidence (green/amber/red).
    - Why: Efficient incremental chart update.
    - Dependencies: steps 16, 17
    - Risk: Low

19. **Implement `selectObject`** (`frontend/src/residuals.js`)
    - Action:
      1. Set `chartState.selectedNoradId = noradId`.
      2. Update the title text to `"Object: " + noradId` (or "No object selected" if null).
      3. If `noradId` is not null and `dataStore.has(noradId)`, call `_redrawChart(chartState, noradId)`.
      4. If `noradId` is null or no data exists, clear the chart lines and show a "No data" message.
    - Why: F-056 cross-selection from globe.
    - Dependencies: steps 16, 18
    - Risk: Low

20. **Implement `renderNoiseBand` and `renderNisThreshold`** (`frontend/src/residuals.js`)
    - Action: These are already called during `_redrawChart` internally. The exported functions allow `main.js` to update the thresholds if needed (e.g., if the backend provides per-object sigma values). For now, `renderNoiseBand` updates the band area data, and `renderNisThreshold` repositions the threshold line (default: 12.592).
    - Why: F-054 reference envelope.
    - Dependencies: step 16
    - Risk: Low

### Phase 5: Anomaly alert panel (`alerts.js`)

21. **Implement `initAlertPanel`** (`frontend/src/alerts.js`)
    - Action:

## Risks and mitigations

- **Risk:** GMST computation error causes all satellite positions to be rotated to incorrect longitudes. **Mitigation:** Validate against Vallado reference values in a standalone test page before integrating. The ISS is at a known inclination (~51.6 degrees) and should visually appear in the correct latitude band.

- **Risk:** CesiumJS Entity API performance degrades with 50 objects plus 50 ellipsoids (100 entities total). **Mitigation:** NF-003 requires 30 FPS with 50 objects. CesiumJS handles thousands of entities; 100 is well within limits. If ellipsoids are costly, they can be rendered as translucent billboards instead (visual downgrade but performance safe).

- **Risk:** The WebSocket message schema does not include a raw residual vector, only NIS and the state estimate. The "residual magnitude" chart (F-054) cannot display the actual residual. **Mitigation:** Use the RSS of covariance diagonal square roots as a "current uncertainty" proxy for the top chart. NIS is the primary metric anyway. If the reviewers require actual residual magnitude, the backend schema must be extended (see open question 1).

- **Risk:** The `GET /catalog` endpoint may not return full state vectors needed for NF-012 reconnection sync. **Mitigation:** Check the main.py plan and implementation; if the response is summary-only, propose a schema extension (see open question 3).

- **Risk:** The WebSocket message schema does not include object name, only `norad_id`. F-055 requires "object name" in the alert feed. **Mitigation:** Fetch the catalog at startup and build a `norad_id -> name` lookup map. Use this map in `alerts.js`. If name is unavailable, fall back to displaying the NORAD ID (see open question 2).

- **Risk:** Demo instability from WebSocket disconnection during the 4-hour session (NF-004). **Mitigation:** Exponential backoff reconnection in `main.js` with automatic full-state resync on reconnect (step 27).
